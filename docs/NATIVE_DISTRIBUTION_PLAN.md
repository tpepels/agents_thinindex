# Native Distribution Plan

This plan distinguishes the shipped archive release path from future signed
native distribution. It is a planning and release-governance document only.

## Current Shipped Distribution

The implemented release path is a local archive:

- Linux/macOS-style `.tar.gz` archives with Unix install/uninstall helpers.
- Windows `.zip` archives with PowerShell install/uninstall helpers.
- SHA256 checksum sidecars.
- `SBOM.md` and `THIRD_PARTY_NOTICES` in the archive.
- Archive content checks and packaged binary smoke checks.

Archive RC readiness does not mean production native distribution readiness.
Unsigned archives are useful for local validation and limited handoff, but they
are not signed native packages, notarized installers, package-manager releases,
or managed update channels.

| winget | None | Manifest submission after signed Windows release | Deferred |

## External Credentials And Tools

Production signing/notarization needs external accounts, tools, or secrets that
are not committed and are not required for the current archive path.

| Area | Required credentials/tools | Blocker |
| --- | --- | --- |
| Windows Authenticode | Code-signing certificate, certificate password, timestamp URL, `signtool` | External certificate and Windows signing host |
| macOS Developer ID | Apple Developer membership, Developer ID identity, team ID, `codesign` | External Apple account and macOS signing host |
| macOS notarization | Apple notarization profile or API credentials, `xcrun notarytool` | External Apple notarization credentials and macOS host |
| Linux package signing | GPG signing key, `gpg`, package repository policy | External signing key and repository policy |
| Homebrew | Tap ownership, formula review, release URL/checksum policy | External publishing account/review |
| winget | Signed Windows artifact, manifest metadata, Microsoft review/submission path | External account/review and signed artifact |
| Update channels | Release hosting, signature verification policy, rollback policy | Product/security design not implemented |

`scripts/sign-release-artifact` is only a scaffold for local or CI signing when
the required environment and tools exist. It is not part of the default archive
build and does not make an artifact production-signed by itself.

Windows and macOS production signing require generated signable native artifacts
or app/package payloads. Running the scaffold against the current archive RC
does not replace MSI/MSIX, `.pkg`/`.dmg`, Developer ID, notarization, stapling,
or package install smoke work.

## Platform Smoke Requirements

Target artifacts are not publishable until the exact archive or package passes
smoke on a compatible platform:

- Linux x86_64 archives on Linux x86_64.
- Linux ARM64 archives on Linux ARM64.
- macOS Intel archives/packages on macOS Intel.
- macOS Apple Silicon archives/packages on Apple Silicon.
- Windows x64 archives/packages on Windows x64.

The target smoke checklist is maintained in `docs/TARGET_PLATFORM_SMOKE.md`.
Cross-compilation or archive naming is not publishable evidence.

## Package Manager Path

Archive publishing and package-manager publishing are separate tracks.

Before Homebrew, winget, Linux repositories, or other package managers are
claimed:

- the native artifact must be generated by a documented script;
- the artifact must pass content checks and target-platform smoke;
- signing/notarization status must be explicit for that platform;
- package metadata must include `THIRD_PARTY_NOTICES`, license, checksum, and
  uninstall/rollback notes;
- the required production SBOM format must be decided, including whether the
  compact archive `SBOM.md` is enough or whether SPDX/CycloneDX output is
  required by the package manager or distribution channel;
- publishing must use an account/review process documented outside the archive
  RC checklist.

No current release workflow publishes GitHub Releases, Homebrew formulas,
winget manifests, Linux repositories, app stores, or update feeds.

## Production Distribution Blockers

The following block production signed/native distribution:

- no generated `.deb`, `.rpm`, AppImage, MSI, MSIX, `.pkg`, or `.dmg` artifacts;
- no completed Authenticode signing;
- no completed macOS Developer ID signing, notarization, or stapling;
- no Linux package signing or repository metadata;
- no Homebrew or winget publishing workflow;
- no managed update channel or rollback policy;
- no target-platform smoke evidence for most non-Linux archive targets;
- no CI secret policy for signing credentials;
- no production SBOM-format decision for package-manager/native distribution;
- no release-token policy for package-manager or GitHub Release publishing.

These blockers do not block the local archive RC path when the archive checks,
checksum sidecar, SBOM/notices checks, and target-platform smoke are green.
An archive without a generated and verified `.sha256` sidecar is not
release-ready.

## Staged Roadmap

1. Keep archive RC release gates green: `scripts/check-release`,
   `scripts/check-package-contents`, `scripts/smoke-release-archive`, checksum
   verification, SBOM/notices inclusion, and target-platform smoke.
2. Add native package generation for one platform at a time, starting with the
   format that has a maintainer-owned test platform and signing path.
3. Add signing/notarization for that platform using CI/local secrets, never
   committed files.
4. Add native package content checks, install/uninstall smoke, and rollback
   notes for that platform.
5. Add package-manager publishing only after signed artifacts, checksums,
   notices, and review metadata are stable.
6. Design update channels separately, including signature verification,
   downgrade/rollback behavior, and failure recovery.

## Rollback And Uninstall

Current archive uninstall helpers remove installed thinindex command files from
the chosen bin directory only. They do not delete `.dev_index`,
`.thinindexignore`, `AGENTS.md`, `CLAUDE.md`, Cursor rules, Copilot
instructions, or project files.

Future native packages must document:

- what files are installed;
- what files uninstall removes;
- how PATH or shell integration is handled;
- how a user downgrades to a previous version;
- how a broken update is rolled back without deleting repository-local caches.
